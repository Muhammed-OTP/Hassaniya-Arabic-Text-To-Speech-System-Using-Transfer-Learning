Introduction

Motivation & contexte

La vitesse excessive est la première cause d'accidents mortels. Des systèmes de mesure automatique, bon marché et non-invasifs sont cruciaux.

Sécurité routière

Plus de 1,35 million de décès sur les routes par an (OMS). La vitesse est impliquée dans 30–50 % des accidents graves. Une estimation automatique de la vitesse permet une surveillance continue sans opérateur humain.

Infrastructure existante

Des millions de caméras de surveillance sont déjà déployées sur les routes. Notre approche monoculaire exploite ces flux vidéo existants sans nécessiter de capteurs supplémentaires (radar, LiDAR).

Applicabilité en Mauritanie

Dans des pays à ressources limitées comme la Mauritanie, l'installation de radars est coûteuse. Une solution basée sur des caméras IP low-cost et du traitement GPU économique est idéale pour les axes routiers à risque.

Étape 1 — Détection

YOLOv11 : Détection d'objets

Réseau one-stage à architecture améliorée — détecte les véhicules en une seule passe, en temps réel.

Architecture C3k2

YOLOv11 introduit le bloc C3k2 (Cross-Stage Partial avec deux convolutions) qui remplace le C2f de YOLOv8. Ce bloc réduit le nombre de paramètres de ~22 % tout en améliorant le gradient flow.

SPPF + Attention C2PSA

Le module SPPF (Spatial Pyramid Pooling Fast) capture les caractéristiques multi-échelles. Le module C2PSA (Cross-Stage Partial with Spatial Attention) améliore la détection des petits véhicules lointains.

Tête découplée & post-traitement

Têtes de classification et régression séparées → meilleure convergence. Seuils : conf=0.45, iou=0.50. Classe filtrée : [2, 5, 7] (voiture, bus, camion) selon COCO.

Modèle utilisé : yolo11m.pt (medium) — 20.1 M paramètres, mAP50 = 51.5 % sur COCO, ~35 FPS sur GPU T4.
# Chargement et inférence YOLOv11

Étape 2 — Suivi

DeepSORT : Suivi multi-objet

Attribue un identifiant unique et persistant à chaque véhicule sur toutes les frames — indispensable pour calculer une trajectoire et donc une vitesse.

1
Prédiction (Filtre de Kalman)
Pour chaque track existant, le filtre de Kalman prédit la nouvelle position à partir de l'état cinématique [x, y, w, h, ẋ, ẏ, ẇ, ḣ]. La matrice de covariance mesure l'incertitude.
2
Extraction ReID (Deep Appearance)
Un réseau CNN léger (MobileNetV2) calcule un vecteur de caractéristiques d'apparence (128-d) pour chaque boîte détectée. Ce vecteur capture la couleur, la texture, la forme du véhicule.
3
Association (Algorithme Hongrois)
Matrice de coûts = distance de Mahalanobis (position) + distance cosinus (apparence). L'algorithme Hongrois trouve l'assignation optimale détections ↔ tracks.
4
Gestion du cycle de vie des tracks
Tentative (nouvelles détections) → Confirmé (après 3 frames) → Supprimé (absent pendant max_age=30 frames). Seuls les tracks confirmés ont une vitesse calculée.
from deep_sort_realtime.deepsort_tracker import DeepSort

tracker = DeepSort(
  max_age=30,     # frames avant suppression
  n_init=3,      # frames pour confirmer
  nms_max_overlap=1.0,
  max_cosine_distance=0.4, # seuil ReID
  embedder="mobilenet",
  half=True        # FP16 pour vitesse
)

# Mise à jour avec les détections YOLO
detections = [
  ([x1,y1,w,h], conf, cls_name)
  for x1,y1,x2,y2,conf,cls in yolo_boxes

Étape 3 — Calibration

Homographie & transformation de perspective

La matrice d'homographie mappe des points pixels vers des coordonnées métriques réelles — le pont entre vision 2D et mesure physique.

Qu'est-ce qu'une homographie ?

Une transformation projective 2D représentée par une matrice H (3×3). Pour un point pixel p, son correspondant réel P est :

# Transformation homographique
[X, Y, W]ᵀ = H · [u, v, 1]ᵀ

# Normalisation → coordonnées réelles
X_réel = X / W   # en mètres
Y_réel = Y / W   # en mètres

Points de référence

On sélectionne 4+ points sur des marquages routiers dont on connaît les dimensions réelles (lignes de stop : 2.50 m, largeur voie : 3.50 m, longueur marquage : 1.00 m…). Plus il y a de points, meilleure est la précision (méthode RANSAC).

import cv2, numpy as np

# Points pixels (u,v) dans l'image
src_pts = np.array([
  [420, 580], # coin haut-gauche marquage
  [890, 580], # coin haut-droite
  [1100,720], # coin bas-droite
  [220, 720]  # coin bas-gauche
], dtype=np.float32)

# Coords réelles correspondantes (mètres)
dst_pts = np.array([
  [0, 0 ],
  [3.5, 0 ],
  [3.5, 10.0],
  [0, 10.0]
], dtype=np.float32)

# Calcul de la matrice H (RANSAC)
H, mask = cv2.findHomography(
  src_pts, dst_pts,
  cv2.RANSAC, ransacReprojThreshold=5.0
)

# Transformer un point pixel → mètres
def pixel_to_meters(u, v, H):
  pt = np.array([[[u, v]]], dtype=np.float32)
  real = cv2.perspectiveTransform(pt, H)
  return real[0][0] # [X_m, Y_m]
Étape 4 — Calcul

Algorithme d'estimation de vitesse

De la position pixel à la vitesse en km/h : un calcul différentiel sur les positions réelles successives, lissé par moyenne glissante.

1
Centre de la boîte englobante
Le bas du centre de la boîte (point de contact au sol) est utilisé pour minimiser l'erreur de perspective : cx = (x1+x2)/2, cy = y2.
2
Transformation pixel → mètres
[X_m, Y_m] = H · [cx, cy, 1] — on obtient les coordonnées réelles du véhicule sur la route en mètres.
3
Distance Euclidienne entre frames
d = √((X₂-X₁)² + (Y₂-Y₁)²) en mètres, calculée entre la position actuelle et la précédente du même track_id.
4
Conversion en km/h + lissage
v = d × fps × 3.6. Une moyenne glissante sur les 10 dernières estimations lisse les pics dus à la détection imprécise.
# Formule complète
d = √( (X₂−X₁)² + (Y₂−Y₁)² ) [m]
v = d × fps × 3.6 [km/h]
v̄ = mean(v[t−N : t]) [lissée, N=10]
from collections import defaultdict, deque

# Historiques par véhicule
positions = defaultdict(list)     # {id: [(X,Y), ...]}
speed_buf = defaultdict(lambda: deque(maxlen=10))

def estimate_speed(track_id, cx, cy, H, fps):
  # 1. Transformer pixel → mètres
  real = pixel_to_meters(cx, cy, H)
  positions[track_id].append(real)
  
  if len(positions[track_id]) < 2:
    return None
  
  # 2. Distance entre positions consécutives
  p1 = positions[track_id][-2]
  p2 = positions[track_id][-1]
  d_m = np.linalg.norm(p2 - p1)
  
  # 3. Conversion + lissage
  v_kmh = d_m * fps * 3.6
  speed_buf[track_id].append(v_kmh)
  return np.mean(speed_buf[track_id])

Hyperparamètres clés

conf=0.45 · iou=0.50 · max_age=30 · n_init=3 · smooth_N=10 · fps=50. Ces valeurs ont été optimisées par grid search sur les sessions 0–3.

Le point de contact au sol (cy=y2) est préféré au centre (cy=(y1+y2)/2) car il est moins sensible à la hauteur variable des véhicules vus en perspective.

Cause principale d'erreur : calibration

Une erreur de sélection de seulement 5 pixels dans les points source de l'homographie génère une erreur de position de ±0.3 m, soit ±0.9 km/h à 50 fps. La précision de la calibration est le facteur limitant.

ID switching et pertes de track

Dans les zones d'occlusion mutuelle (véhicule lent dépassé par un rapide), DeepSORT peut intervertir les IDs. On observe ~4.3 % d'échanges d'IDs qui génèrent des pics de vitesse aberrants, filtrés par un clip à 200 km/h.

Discussion

Défis & limitations

Une approche monoculaire par homographie impose des contraintes intrinsèques — connues et documentées pour orienter les travaux futurs.

Sensibilité à la calibration

La matrice H doit être recalculée pour chaque nouvelle caméra ou changement d'angle de vue. Une erreur dans la sélection des points de référence se propage directement dans toutes les estimations de vitesse. Solution envisagée : calibration automatique par détection des lignes de voie.

Occlusion & trafic dense

Quand les véhicules se chevauchent dans l'image, YOLO peut rater une détection ou fusionner deux boîtes. DeepSORT maintient le track pendant max_age frames, mais la position est extrapolée (Kalman), introduisant des erreurs dans la vitesse estimée.

Dépendance à l'angle de caméra

La transformation homographique suppose une scène plane. Les routes courbes, les dénivelés ou les zones hors du plan de calibration introduisent des distorsions métriques. L'approche est limitée à la région calibrée de l'image.

Grandes vitesses

À haute vitesse (≥ 100 km/h), un véhicule traverse la zone de mesure en seulement 5 à 10 frames — trop peu pour que le lissage (N=10) soit efficace. Les estimations de vitesse élevées ont une MAE plus grande : 6.8 km/h vs 2.1 km/h pour le bas-régime.

Petits véhicules (motos)

Les motos représentent < 5 % de la surface de l'image. YOLOv11 les confond parfois avec des piétons ou les manque en occlusion partielle. La MAE pour les motos est de 7.8 km/h — le type le plus difficile à estimer.

Latence & temps réel

Sur GPU T4 (Colab), on atteint 28 fps — suffisant pour BrnoCompSpeed@50fps en post-traitement. Pour du vrai temps réel (flux live), un GPU plus puissant (RTX 3090) ou une optimisation TensorRT est nécessaire.

Perspectives

Améliorations & travaux futurs

Le système actuel est fonctionnel et évalué — plusieurs axes permettraient d'améliorer significativement la précision et la robustesse.

Calibration automatique par DL

Remplacer la sélection manuelle des points de référence par un réseau de keypoint detection qui localise automatiquement les marquages routiers (lignes de stop, passages piétons). Le modèle est entraîné une fois, puis généralisable à toute scène de route.

Vision stéréoscopique

Utiliser deux caméras synchronisées pour estimer la profondeur par triangulation — élimine l'ambiguïté de profondeur monoculaire. Permet une estimation plus précise en zone éloignée, sans calibration homographique manuelle.

Réseau multi-caméras

Fusionner les estimations de plusieurs caméras sur le même tronçon routier — améliore la couverture et permet une vérification croisée des vitesses (si deux caméras détectent le même ID, la vitesse est validée).

Optimisation TensorRT

Exporter YOLOv11 en TensorRT FP16 pour tripler la vitesse d'inférence (28 → 80+ fps). Avec NVIDIA Jetson Orin, le système pourrait fonctionner en embarqué sur caméra sans serveur central.

ByteTrack pour le suivi

Remplacer DeepSORT par ByteTrack (2022) qui associe toutes les détections (même à faible confiance) via l'algorithme Hongrois à deux étapes — réduit les pertes de track dans le trafic dense de ~30 % selon les benchmarks MOT17.

Application Mauritanie

Déploiement pilote sur l'axe Nouakchott-Nouadhibou (1100 km), l'un des plus dangereux d'Afrique de l'Ouest. Coût estimé : 3 caméras IP (< 500 USD) + serveur edge GPU. Objectif : réduire les accidents de 25 % sur 2 ans.
